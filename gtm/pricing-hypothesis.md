# Pricing Hypothesis

Status: Draft
Purpose: Early pricing discussion and pilot validation
Important: These are hypotheses, not final prices.

## Pricing principle

SpecBridge AI should be priced based on tender review value, report package value and team productivity gain, not as a generic document parser.

## Possible pricing models

### Model A: Per-document analysis

Best for early pilots and low-volume customers.

Example hypothesis:

| Tier | Monthly document volume | Positioning |
|---|---:|---|
| Starter | 10 documents/month | Small teams |
| Professional | 50 documents/month | Active presales teams |
| Business | 150 documents/month | Larger bid teams |

### Model B: Per-seat subscription

Best when multiple users need access to the same analysis workflow.

Possible seats:

- Viewer
- Analyst
- Admin
- Manager

### Model C: Team package

Best for system integrators and vendor country teams.

Package may include:

- Monthly document quota
- Multiple report profiles
- Export formats
- Pilot onboarding
- Custom templates
- Priority support

### Model D: Pilot package

Best for first 3 to 5 pilot customers.

Suggested pilot structure:

- Fixed 30 or 60 day pilot
- Limited number of anonymized documents
- Structured feedback sessions
- No production commitment
- Clear data deletion rule

## Early recommendation

Start with a controlled pilot package, then test team-based pricing.

Avoid publishing exact pricing on the website until value perception and usage pattern are validated.

## Value metric candidates

- Number of analyzed tender documents
- Number of generated report packages
- Number of users
- Number of report profiles
- Export format access
- Custom template support
- Private deployment / local mode

## Pricing risks

- Underpricing as simple AI summarization
- Overpricing before trust is established
- Not separating pilot pricing from production pricing
- Ignoring security/KVKK requirements for enterprise buyers
